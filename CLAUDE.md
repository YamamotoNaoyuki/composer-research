# Composer Research Assistant

作曲家リサーチを体系的に進めるためのCLIツール群。クラシック音楽の知識を各作曲家について効率よく構築し、「自分の言葉で語れる」レベルに到達することが目的。

## プロジェクトの背景

ショパンやグリーグのように伝記を読み、多くの曲を聴いてきた作曲家もあれば、名前と数曲しか知らない作曲家もある。このプロジェクトは「知らない作曲家を効率よく知る」プロセスを、学術的に信頼性の高いソースに基づいて支援する。

## 対象とするリサーチの2段階

### 第1段階：骨格をつかむ（Skeleton）

作曲家について「30分で語れる」状態を作る。以下の項目を構造化して収集・整理する：

- **基本情報**: 生没年、出身地、主な活動都市
- **時代背景**: その作曲家が生まれた年がどういう時代で、どういう歴史的背景のもとに生きたかを記述する。政治体制、社会情勢、文化潮流、同時代の主要な歴史的事件を含める。その作曲家の人生と創作を理解するための前提知識として機能させる。
- **音楽史上の位置づけ**: 時代区分、様式・流派、影響を受けた人物と与えた人物
- **年表**: その作曲家の一生を年表だけで理解できるレベルの詳細さで作成する。年齢は全行に必ず明記する。含めるべき項目は：
  - 生い立ち・家庭環境・音楽教育の開始
  - 師弟関係・重要な出会い（パトロン、共演者、対立した人物、恋愛関係）
  - 転居・活動拠点の変遷（都市名と理由）
  - 作風の転換点（初期/中期/後期の区分とそのきっかけ）
  - 主要作品の作曲・初演年（代表作リストとクロスリファレンス）
  - 経済状況の変化（収入源、パトロン、生活水準）
  - 健康状態の推移
  - 同時代の歴史的事件との関連（革命、戦争、万博など）
  - 晩年・死因・死後の評価の変遷
  年表は「事実の羅列」ではなく「なぜその時期にその作品が生まれたか」を理解するための骨組みとして機能させる。年表を読むだけでその作曲家の人生の全体像が浮かぶことを目指す。
- **代表作**: 5〜10曲。ジャンル横断で選ぶ（交響曲だけ、ピアノ曲だけにならないように）
- **定番エピソード**: その作曲家を語るとき言及されるエピソードをできる限り多く収集する。数の制限なし。
- **ソース**: 各項目に対して使用したソースとその信頼性ランク（品質基準セクション参照）を記録する

### 第3段階：語れるようにする（Articulation）

知識を「自分の言葉で説明できるか」テストする。以下の3問を軸にする：

1. **一言で言うと？** — この作曲家の音楽の本質を一文で表現する
2. **同時代との差異は？** — 同時代の作曲家と比較して何が独自か
3. **最初の1曲は？** — 初めて聴く人への推薦曲とその理由

回答に詰まった箇所 = 知識のギャップとして記録し、追加リサーチの対象にする。

## 技術スタック

- **実行環境**: Claude Code — リサーチ、Web検索、ファイル生成、Articulationの壁打ちをすべて Claude Code セッション内で直接実行する。外部 API キー不要。
- **Web検索**: Claude Code の WebSearch ツール — 事実確認と最新情報の取得
- **データ保存**: ローカルMarkdownファイル + JSON メタデータ
- **バージョン管理**: GitHub リポジトリ（private）で全データ・コードを管理
- **補助ツール**: TypeScript（ts-node）— ティア管理などのユーティリティスクリプト用

## 運用フロー

すべての作業は Claude Code セッション内で直接実行する。TypeScript スクリプトによる API 呼び出しは不要。

```

# 第3段階：Articulationテスト

npx ts-node src/utils/tier.ts list               # ティア別作曲家一覧

```

## 作曲家ティアの定義

- **Tier 1**: 最優先で学ぶ作曲家。第3段階まで完了を目指す。
- **Tier 2**: 次に学ぶ作曲家。まず第1段階を完了し、第3段階は段階的に。
- **Tier 3**: 将来取り組む。第1段階のみで当面は十分。

ティアの割り当ては、自分の既存知識レベル × 興味の強さ × クラシック音楽全体の見通しに必要かどうか で決める。

## リサーチ方針

- Skeleton 生成時は WebSearch でソース信頼性ポリシー（品質基準セクション参照）に従った情報取得を徹底する。各事実に対してソースのランクを明示し、ランク3以下のみの情報には「要裏取り」フラグをつける。
- 年表生成では、単なる年号の羅列ではなく「作風の転換点」「なぜその時期にその作品が書かれたか」という因果関係を含める。同時代の歴史的事件との関連も拾う。
- Articulation テストでは、ユーザーの回答に対して「答えの正しさ」だけでなく「他人に説明できるレベルの深さがあるか」を評価する。
- 日本語で回答する。ただし作品名・人名は原語表記を併記する。

## Claude Code セッション運用

- 作業開始時に PROGRESS.md を確認し、前回の続きから始める。
- 新しい作曲家のリサーチを始めたら PROGRESS.md に日付とともに記録する。
- 1セッションで1〜2人の作曲家に集中する。広く浅くやらない。
- kage-bunshin の `music/` ドメインフォルダにも要約を反映する。
- セッション終了時には必ず変更をコミット＆プッシュする（下記 Git 運用ルール参照）。

### コミット粒度とメッセージ規約

**ひとつの変更ごとに即 commit & push する。** リモートセッション（iPhone / Termius）からスマホで進捗を追えるようにするため、変更を溜めない。

「ひとつの変更」の単位：
- ファイル1つの新規作成・更新が1コミット
- 複数ファイルを同時に変更する場合でも、論理的に分離できるなら分ける
- ただし1つのコマンド実行で複数ファイルが同時生成される場合（例: skeleton.ts が skeleton.md と composers.json を同時に更新）はまとめてよい

```bash
# 例: skeleton 生成 → 即 push
git add data/composers/debussy/skeleton.md data/composers.json
git commit -m "skeleton: Debussy — 初回AI生成"
git push

# 例: 年表の1項目を修正 → 即 push
git add data/composers/debussy/skeleton.md
git commit -m "skeleton: Debussy — 年表修正（パリ移住年を1831に訂正）"
git push

# 例: Articulationテスト → 即 push
git add data/composers/debussy/articulation.md
git commit -m "articulation: Debussy — 初回テスト、ギャップ3件"
git push

# 例: コード変更 → 即 push
git add src/skeleton.ts
git commit -m "feat: skeleton.ts に年表生成ロジック追加"
git push

# 例: PROGRESS.md 更新 → 即 push
git add PROGRESS.md
git commit -m "docs: PROGRESS.md 更新"
git push
```

接頭辞: `skeleton:` / `articulation:` / `gap:` / `data:` / `feat:` / `fix:` / `docs:`

### Claude Code セッションでの Git 操作

1. **セッション開始時**: `git pull` で最新化
2. **変更のたびに**: `git add` → `git commit` → `git push` を即実行。変更を手元に溜めない
3. **セッション終了時**: `git status` で未コミットの変更がないことを確認

コード変更（src/）とデータ変更（data/）は別コミットにすること。履歴の関心が異なるため。

## 品質基準

### ソース信頼性ポリシー（最重要）

情報の取得元には明確な優先順位をつける。上位のソースが使える場合は下位に頼らない。

1. **最優先**: Grove Music Online（Oxford Music Online）、MGG Online、学術書・研究書（大学出版局刊行のもの）
2. **信頼できる**: IMSLP の作品情報、New Grove Dictionary of Music（書籍版）、各国の音楽百科事典（日本では『ニューグローヴ世界音楽大事典』）
3. **補助的に使える**: 英語版 Wikipedia の B+クラス以上の記事（ただし必ず上位ソースとクロスチェック）、Allmusic のバイオグラフィー
4. **使わない**: 個人ブログ、SNS、出典不明のまとめサイト、日本語 Wikipedia（品質のばらつきが大きいため一次ソースにしない）

Skeleton 生成時の WebSearch 結果は上記基準でフィルタリングする。検索対象ドメインを信頼できるソースに限定する。

### その他の品質基準

- skeleton.md の事実情報は最低1つの信頼できるソース（上記ランク1〜2）で裏取りされていること。
- 年表の年号・都市名は特に誤りが混入しやすい。AIが自信ありげに出す年号ほど要確認。作品番号（Op.）と作曲年の対応も検証する。
- AIが生成した「定番エピソード」は特に要注意。俗説・誤伝の混入率が高い。
- 代表作リストは「有名な曲」だけでなく「その作曲家の本質を示す曲」を含めるよう意識する。
